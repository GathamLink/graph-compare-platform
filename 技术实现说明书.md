## 1. 项目定位与场景

本工具的核心需求：**构建一个通用的图片对比平台，让需要对比图片差异的用户，能够快速上传、自动配对并获得量化的差异评估结果。**

区别于简单的"肉眼对比"，本平台解决的问题是：
- **量化**：给出 0~100% 的相似度分数，而不是"看起来差不多"
- **定位**：自动标注差异区域（红框标记），不需要逐像素人工排查
- **批量**：一次处理几十对甚至上百对图片，结果统一管理

### 典型使用场景

| 场景 | 说明 |
|------|------|
| **文档导出差异检测** | 同一份 PDF / Word 在不同平台（Windows/macOS/在线工具）导出为图片，对比渲染差异（字体、间距、颜色） |
| **UI 改版对比** | App 或网页迭代前后的截图对比，快速找出变更的区域 |
| **图像处理验证** | 图像处理算法的输入/输出对比，确认处理效果是否符合预期 |
| **多设备渲染一致性** | 同一内容在不同设备/分辨率/DPI 下的渲染结果对比 |
| **版本管理** | 任何需要对比"两组相关图片"差异的场景 |

### 核心问题

设计本平台需要解决三个核心问题：

1. **如何组织"两组图片一一配对"的数据结构？**
   → 用任务（Task）作为容器，A/B 两组图片按 `sort_order` 排序后位置对应即为一对，支持拖拽重排和前缀自动匹配

2. **如何自动计算图片相似度，不依赖人工判断？**
   → 信息区掩码 + 四项指标融合评分（详见第 3.4 节），给出 0~100% 的量化结果和差异区域标注

3. **如何在前端高效展示大量图片对比？**
   → 缩略图列表 + 点击弹窗详情，支持并排/滑块两种对比模式，键盘快捷键导航

## 2. 整体架构

### 2.1 技术选型与理由

| 层级 | 选择 | 理由 |
|------|------|------|
| **前端** | React 18 + TypeScript + Vite | 生态成熟，TypeScript 减少低级错误，Vite 热更新快 |
| **UI 组件** | shadcn/ui + Tailwind CSS | 组件按需引入，不引入全局样式污染；Tailwind 避免起类名 |
| **状态管理** | Zustand | 比 Redux 轻量 10 倍，不需要 Provider 包裹，适合中小型应用 |
| **后端** | Python FastAPI | 自动生成 OpenAPI 文档；原生 async/await；Pydantic 数据校验 |
| **Python 管理** | uv | 比 pip 快 10~100 倍；lockfile 管理；虚拟环境自动创建 |
| **数据库** | SQLite + SQLAlchemy | 零配置，单文件存储，适合本地部署；ORM 避免 SQL 注入 |
| **对象存储** | MinIO | 兼容 AWS S3 API，本地部署无需云服务；前端可直连取图，减轻后端压力 |
| **图像处理** | OpenCV + scikit-image | SSIM 算法来自 scikit-image，轮廓检测用 OpenCV，Pillow 生成缩略图 |

### 2.2 核心流程

```
用户操作流程：
  创建任务 → 上传图片（手动/ZIP）→ 触发对比计算 → 查看结果列表 → 点击查看详情 → 导出报告

后台计算流程：
  上传完成 → BackgroundTasks 异步触发
      → 遍历所有配对
      → 跳过已有缓存（diff_results 表唯一索引）
      → 对每对图片：对齐 → 掩码 → 计算四项指标 → 融合评分
      → 生成差异标注图 → 上传 MinIO
      → 写入 diff_results 表

数据流：
  图片文件 → MinIO（images/xxx.png）
  缩略图  → MinIO（thumbs/xxx.webp）
  差异图  → MinIO（diffs/xxx_diff.png）
  元数据  → SQLite（images 表存 oss_key，不存路径）
```

### 2.3 架构设计关键决策

**决策 1：图片不存数据库，只存 OSS Key**

```python
# ❌ 错误做法：存文件路径（换机器就失效）
filepath = "/home/user/uploads/abc.png"

# ✅ 正确做法：存 MinIO 对象 Key
oss_key = "images/abc123.png"
# 访问 URL 动态拼接：http://localhost:13012/graph-compare/{oss_key}
```

**决策 2：配对关系用 sort_order 决定，而非硬编码**

```
A 组：[sort_order=0, sort_order=1, sort_order=2]
B 组：[sort_order=0, sort_order=1, sort_order=2]
            ↕ 配对              ↕ 配对        ↕ 配对
```

拖拽排序后重新写入 sort_order，配对关系随之改变，无需修改其他逻辑。

**决策 3：前端直连 MinIO 取图，不走后端**

```
❌ 低效：浏览器 → FastAPI → MinIO → FastAPI → 浏览器（两次传输）
✅ 高效：浏览器 → MinIO（直连，一次传输）
         FastAPI 只负责业务逻辑，不传输图片字节
```

### 3.2 图片上传与缩略图

**缩略图生成策略**：上传时同步生成 200px 宽的 WEBP 缩略图。列表页展示缩略图（5~15KB），对比弹窗展示原图（几百KB~几MB），大幅减少列表页加载时间。

```python
from PIL import Image as PILImage, ImageOps
import io

THUMB_SIZE = 200  # 缩略图最大宽度（px）

def _generate_thumbnail(image_bytes: bytes, mime_type: str) -> bytes | None:
    """生成 200px 宽的 WEBP 缩略图，失败静默返回 None"""
    try:
        PILImage.MAX_IMAGE_PIXELS = None  # 允许超大图片（如高分辨率 PDF 导出）
        img = PILImage.open(io.BytesIO(image_bytes))
        img = ImageOps.exif_transpose(img)  # 修正手机拍照的 EXIF 旋转
        img = img.convert("RGB")           # 统一转 RGB（处理 RGBA PNG）

        w, h = img.size
        if w > THUMB_SIZE:
            new_h = int(h * THUMB_SIZE / w)
            img = img.resize((THUMB_SIZE, new_h), PILImage.LANCZOS)

        buf = io.BytesIO()
        img.save(buf, format="WEBP", quality=80)
        return buf.getvalue()
    except Exception:
        return None  # 静默失败，不影响主流程
```

**为什么选 WEBP**：同等质量下比 PNG/JPEG 小 25~35%，现代浏览器全支持。

### 3.3 ZIP 批量导入

**ZIP 目录结构规范**：
```
upload.zip
  └── root/（名称随意）
      ├── 原图/
      │   ├── 001.png
      │   └── 002.png ...
      └── 对比图/
          ├── 001.png
          └── 002.png ...
```

**最大坑：macOS zip 的中文文件名乱码**

Python 的 `zipfile` 模块默认用 CP437 解码文件名，中文 UTF-8 字节会被误解码成乱码：

```python
# "原图" 的 UTF-8 编码被 CP437 误读 → "σÄƒσ¢╛"（乱码）
```

**解决方案**：

```python
def _decode_zip_name(info: zipfile.ZipInfo) -> str:
    """正确解码 ZIP 文件名（兼容 macOS/Windows/Linux 压缩）"""
    # 将 CP437 误读结果反转回原始字节
    raw_bytes = info.filename.encode('cp437')

    # bit11=1 表示 UTF-8 编码（现代工具会设置这个标志）
    if info.flag_bits & 0x800:
        return raw_bytes.decode('utf-8', errors='replace')

    # 否则依次尝试常见中文编码
    for enc in ('utf-8', 'gbk', 'gb2312', 'big5'):
        try:
            return raw_bytes.decode(enc)
        except UnicodeDecodeError:
            continue

    return raw_bytes.decode('utf-8', errors='replace')

# 使用：构建「正确文件名 → ZipInfo」映射
name_map = {_decode_zip_name(info): info for info in zf.infolist()}
```

**目录结构检测**（不依赖目录条目，因为 macOS zip 不生成目录条目）：

```python
EXPECTED = {'原图', '对比图'}
prefix = None
```

### 3.4 图片对比算法

这是整个项目最核心的技术部分。

#### 3.4.1 为什么不用纯 SSIM？

SSIM（结构相似性指数）是图像质量评估的标准算法，但在本场景下有严重局限：

```
问题场景：同一 App 的"账户"页 vs "设置"页
  - 背景色相同 → SSIM ≈ 1.0
  - 布局结构相同 → SSIM ≈ 0.95
  - 文字全换了 → 文字只占 5% 面积，被 95% 相同背景稀释
  - 结果：SSIM = 0.928（92.8%）← 明显虚高，人眼一看就差异很大
```

根本原因：SSIM 是**全局均值指标**，对"小区域大变化"不敏感。

#### 3.4.2 信息区掩码

核心思路：**只在有内容的像素区域内计算差异**，让空白背景不参与评分。

```python
import cv2
import numpy as np

def _info_mask(gray: np.ndarray, lap_threshold: int = 8) -> np.ndarray:
    """
    生成信息区掩码：1=有内容，0=空白背景
    原理：文字/图标的边缘梯度大；纯色背景梯度接近 0
    """
    lap = cv2.Laplacian(gray, cv2.CV_64F)      # 拉普拉斯算子求梯度
    mask = (np.abs(lap) > lap_threshold).astype(np.uint8)

    # 膨胀 7×7：将文字周围 7px 也纳入信息区（捕捉字符间空隙）
    kernel = cv2.getStructuringElement(cv2.MORPH_ELLIPSE, (7, 7))
    return cv2.dilate(mask, kernel)
```

**实测信息区占比**：
- 手机 App 截图（887×1920）：**5.4%** 是内容，94.6% 是背景
- PDF 文档（2481×10524）：**12.8%** 是内容

没有掩码时，94.6% 的"完全相同的背景"会把 5.4% 的内容差异彻底稀释。

#### 3.4.3 四项指标

```python
# 指标 1：掩码差异面积比（权重 0.40）—— 主力指标
# 量化"有多少比例的内容像素变化了"
def _masked_diff_ratio(ga, gb, mask, threshold=15):
    diff = cv2.absdiff(ga, gb)
    info_px = mask > 0
    if info_px.sum() < 100:
        return float(1 - diff.mean() / 255)  # 信息区太少则用全图
    changed = (diff[info_px] > threshold).sum()  # 忽略 JPEG 压缩噪声（thr=15）
    return float(1 - changed / info_px.sum())

# 指标 2：掩码分块 SSIM（权重 0.40）—— 局部结构
# 8×8 分块，跳过空白格，取最差 1/3 的均值
def _masked_grid_ssim(ga, gb, mask, grid=8, worst_frac=1/3):
    h, w = ga.shape
    scores = []
    for i in range(grid):
        for j in range(grid):
            r1,r2 = i*h//grid, (i+1)*h//grid
            c1,c2 = j*w//grid, (j+1)*w//grid
            block_mask = mask[r1:r2, c1:c2]
            if block_mask.mean() / 255 < 0.10:  # 空白块跳过
                continue
            s, _ = ssim(ga[r1:r2, c1:c2], gb[r1:r2, c1:c2], full=True)
            scores.append(max(0.0, float(s)))
    if not scores:
        return 1.0
    scores.sort()
    n = max(1, int(len(scores) * worst_frac))
    return float(np.mean(scores[:n]))  # 最差 1/3 的均值

# 指标 3：全局 SSIM（权重 0.10）—— 辅助
# 权重低，只作为"是否同一场景"的辅助判断

# 指标 4：HSV 颜色直方图（权重 0.10）—— 颜色主题
# 对颜色主题切换（深/浅色模式）敏感
```

**为什么取最差 1/3**：文字替换通常只影响 2~5 个格子，取全部 64 格均值会被 59 个"相同格子"稀释；最差 1/3 聚焦在真正变化的区域。

#### 3.4.4 融合公式

```python
# 加权几何平均（而非算术平均）
# 几何平均的关键特性：任一指标很低，总分都会被大幅拉低
score = (
    masked_diff_ratio ** 0.40
    * masked_grid_ssim ** 0.40
    * global_ssim       ** 0.10
    * hist_similarity   ** 0.10
)

SIMILARITY_THRESHOLD = 0.75
is_similar = score >= SIMILARITY_THRESHOLD
```

**为什么不用算术平均**：
```
场景：diff_ratio=0.1（差异很大），其他指标都=0.9

算术平均：0.4×0.1 + 0.4×0.9 + 0.1×0.9 + 0.1×0.9 = 0.54  → 还凑合
几何平均：0.1^0.4 × 0.9^0.4 × 0.9^0.1 × 0.9^0.1 ≈ 0.36  → 被拉低了
```

几何平均确保"只要有一个维度差异大，总分就低"。

#### 3.4.5 四种算法模式

| 模式 | diff 阈值 | 掩码梯度 | 插值方法 | 适用场景 |
|------|----------|---------|---------|---------|
| `balanced` | 15 | 8 | INTER_AREA | 通用 |
| `document` | 5 | 4 | LANCZOS4 | PDF 文档对比，检测 DPI 差异 |
| `structural` | 20 | 12 | INTER_AREA | 检测布局/元素增删 |
| `pixel_exact` | 3 | — | LANCZOS4 | 验证完全一致，全图无掩码 |

**关键差异**：
- `document` 模式阈值降为 5（PNG 文档无 JPEG 噪声，可以更敏感）
- `pixel_exact` 不使用掩码（每个像素都重要）
- `document` 和 `pixel_exact` 用 LANCZOS4（对文字边缘保留更好，减少缩放伪差异）

## 4. 前端关键实现

### 4.1 对比视图的图片缩放与居中

这是一个常见的坑：**`transform: scale()` 缩放图片，但图片仍占据原始尺寸的空间**，导致滚动异常。

```tsx
// ❌ 错误方案：scale 缩放，占位空间不变
<img style={{ transform: `scale(${zoom})` }} />

// ✅ 正确方案：
{/* 外层：overflow-auto 提供双向滚动 */}
<div className="flex-1 overflow-auto">
  {/* 内层：flex 居中；min-w-full 防止容器塌陷 */}
  <div className="flex flex-col items-center py-4 min-w-full">
    <img style={{
      width: `${zoom * 100}%`,   // zoom=0.25 → 图片25%宽，flex居中显示
      height: 'auto',             // zoom=2.0  → 图片200%宽，出现滚动条
      maxWidth: 'none',           // 覆盖默认的 max-width: 100%
    }} />
  </div>
</div>
```

### 4.2 滑块对比视图的裁剪

```tsx
// 关键：A 图的 img 宽度要等于原始宽度（非裁剪层宽度）
<div style={{ width: `${sliderPos}%`, overflow: 'hidden' }}>  {/* 裁剪容器 */}
  <img style={{
    width: `${10000 / sliderPos}%`,  // 例：sliderPos=50% → img宽200%
    maxWidth: 'none',                 // 这样 img 实际渲染宽度 = 裁剪容器宽度的两倍
  }} />
</div>
// 结果：无论滑块在哪，A 图左半部分始终对齐 B 图
```

### 4.3 大列表性能优化

```tsx

const [expanded, setExpanded] = useState(false)
```

## 6. 踩坑记录

以下是开发过程中遇到的真实问题，复刻时可提前规避。

### 6.1 uv + hatchling 构建失败

**现象**：`uv sync` 报 `Unable to determine which files to ship`

**原因**：`pyproject.toml` 默认用 `hatchling` 作为 build-backend，它要求有与项目同名的子包目录。但我们的后端代码是平铺的（`main.py`、`models.py` 直接在 `backend/` 下）。

**解决**：
```toml
[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.build_meta"

[tool.setuptools]
packages = []    # 明确声明：不打包任何子包
py-modules = []
```

### 6.2 uvicorn 环境下 logging 失效

**现象**：设置了 `logging.basicConfig()`，但日志就是不输出

**原因**：uvicorn 启动时会重置 root logger，覆盖 `basicConfig` 的配置

**解决**：放弃 `logging`，直接写文件：
```python
def diff_log(msg: str):
    today = datetime.date.today().strftime("%Y-%m-%d")
    with open(f"logs/diff.{today}.log", "a", encoding="utf-8") as f:
        ts = datetime.datetime.now().strftime("%Y-%m-%d %H:%M:%S")
        f.write(f"{ts} [DIFF] {msg}\n")
```

### 6.3 FastAPI Query 参数不能用 pattern 校验空字符串

**现象**：前端传 `status=""`，后端返回 422 错误

**原因**：`Query(pattern="^(draft|active)$")` 不允许空字符串，而 Axios 会把 `undefined/null/""` 的参数序列化为 `?status=`

**解决**：
```python
# 后端：手动校验，空字符串视为 None
async def list_tasks(status: Optional[str] = Query(None)):
    status_filter = status.strip() if status else None
    if status_filter and status_filter not in {"draft", "active", "completed"}:
        raise HTTPException(400, "status 参数非法")
```

```typescript
// 前端：发请求前过滤所有空值参数
function cleanParams(params?: Record<string, any>) {
    return Object.fromEntries(
        Object.entries(params ?? {}).filter(([, v]) => v !== undefined && v !== null && v !== '')
    )
}
```

### 6.4 React 组件白屏（useCallback 定义顺序）

**现象**：页面打开白屏，控制台无错误

**原因**：`handleRecompute` 引用了 `stopPoll` 和 `startPoll`，但定义顺序在它们之前，运行时为 `undefined`

**解决**：useCallback 链中，被引用的必须先定义
```typescript
const stopPoll  = useCallback(...)      // 先定义
const startPoll = useCallback(...)      // 再定义（引用 stopPoll）
const handleRecompute = useCallback(...)  // 最后定义（引用 stopPoll/startPoll）
```

### 6.5 macOS ZIP 中文文件名乱码

**现象**：上传 macOS 压缩的 ZIP，中文目录名变成 `σÄƒσ¢╛`，导致找不到目录

**原因**：Python `zipfile` 默认 CP437 解码，中文 UTF-8 字节被误读

**解决**：反转 CP437，再用 UTF-8/GBK 解码（见 3.3 节）
