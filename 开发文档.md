## 1. 项目概述

本平台（**图片对比平台**）用于对比两组图片之间的视觉差异。以"任务"为基本管理单元，每个任务包含两组按顺序排列的图片，平台将两组图片一一配对展示，通过多维度对比算法自动计算相似度，帮助用户快速发现差异。

### 典型使用场景

- PDF / 文档在不同平台导出结果的差异检测
- UI/UX 迭代前后截图对比
- 图像处理算法效果评估
- 同一内容在不同设备/DPI 下的渲染差异

### 核心功能

| 模块 | 功能 |
|------|------|
| 任务管理 | 创建/编辑/删除任务，配置配对方式和对比算法 |
| 图片上传 | 拖拽上传、排序、ZIP 批量导入（原图/ + 对比图/ 目录结构） |
| 对比结果 | 列表展示（正常/异常/待计算徽章）+ 弹窗详情（并排/滑块模式） |
| 差异计算 | 后台异步计算，前端轮询进度，结果缓存复用 |
| 报告导出 | 生成独立 HTML 报告，图片内嵌 Base64（< 50MB）或 MinIO URL（≥ 50MB） |

## 3. 存储服务设计（MinIO OSS）

### 3.1 部署方式（无 Docker）

```bash
# MinIO 二进制由 scripts/install.sh 自动下载
MINIO_ROOT_USER=admin MINIO_ROOT_PASSWORD=password123 \
  ./bin/minio server ./minio-data \
  --address ":13012" --console-address ":13013"
```

### 3.2 Bucket 设计

| Bucket | 目录结构 | 说明 |
|--------|---------|------|
| `graph-compare` | `images/{uuid}.ext` | 原图 |
| | `thumbs/{uuid}.webp` | 缩略图（200px，WEBP，上传时自动生成） |
| | `diffs/{uuid}_diff.png` | 差异标注图 |

> Bucket 设置为内网公开读，前端直连访问：`http://localhost:13012/graph-compare/{oss_key}`

### 3.3 缩略图策略

- 上传图片时同步生成 200px 宽的 WEBP 缩略图（高度等比）
- 列表页、管理页展示缩略图，对比弹窗展示原图
- 旧图片可通过 `scripts/gen_thumbs.py` 批量补生成
- 图片 `thumb_oss_key` 为空时前端自动降级显示原图

## 4. 数据库设计

### 4.1 表结构

#### `tasks` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 任务 ID |
| `name` | VARCHAR(255) | 任务名称 |
| `description` | TEXT | 任务描述 |
| `status` | VARCHAR(20) | `draft` / `active` / `completed` |
| `pair_mode` | VARCHAR(20) | `sequential`（顺序）/ `prefix`（前缀，默认 sequential） |
| `diff_algo` | VARCHAR(20) | `balanced` / `document` / `structural` / `pixel_exact`（默认 balanced）|
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

#### `images` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 图片 ID |
| `task_id` | INTEGER FK | 所属任务 |
| `group` | VARCHAR(1) | `'A'`（原图组）或 `'B'`（对比图组） |
| `sort_order` | INTEGER | 组内排序序号（决定配对关系） |
| `oss_key` | VARCHAR(512) | MinIO 对象 Key（如 `images/abc123.png`） |
| `thumb_oss_key` | VARCHAR(512) | 缩略图 Key（如 `thumbs/abc123.webp`，nullable） |
| `original_name` | VARCHAR(255) | 原始文件名（仅用于展示） |
| `file_size` | INTEGER | 文件大小（字节） |
| `mime_type` | VARCHAR(50) | MIME 类型 |
| `width` | INTEGER | 图片宽度（px） |
| `height` | INTEGER | 图片高度（px） |
| `created_at` | DATETIME | 上传时间 |

> `(task_id, group, sort_order)` 决定配对：A 组第 i 张 ↔ B 组第 i 张

#### `diff_results` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 记录 ID |
| `task_id` | INTEGER FK | 所属任务 |
| `image_a_id` | INTEGER FK | A 组图片 ID（联合唯一索引） |
| `image_b_id` | INTEGER FK | B 组图片 ID（联合唯一索引） |
| `pair_index` | INTEGER | 第几对（0-based） |
| `diff_oss_key` | VARCHAR(512) | 差异标注图 Key |
| `diff_score` | FLOAT | 综合相似度分数（0~1） |
| `is_similar` | BOOLEAN | 是否相似（`diff_score >= 0.75`） |
| `align_method` | VARCHAR(20) | `resize` / `feature` / `none` |
| `size_warning` | BOOLEAN | 尺寸差异 > 10% |
| `status` | VARCHAR(20) | `pending` / `running` / `done` / `failed` |
| `created_at` | DATETIME | 计算时间 |

> `(image_a_id, image_b_id)` 建联合唯一索引，用于增量计算跳过已有缓存。

## 5. 后端 API 设计

**Base URL**：`http://localhost:13011/api/v1`
**API 文档**：`http://localhost:13011/docs`

### 5.1 任务接口

| 方法 | 路径 | 描述 |
|------|------|------|
| `GET` | `/tasks` | 任务列表（分页、搜索、状态筛选） |
| `POST` | `/tasks` | 创建任务（含 pair_mode / diff_algo） |
| `GET` | `/tasks/{id}` | 任务详情（含两组图片列表） |
| `PUT` | `/tasks/{id}` | 更新任务（名称/描述/pair_mode/diff_algo） |
| `DELETE` | `/tasks/{id}` | 删除任务（级联删除图片及 OSS 对象） |

### 5.2 图片接口

| 方法 | 路径 | 描述 |
|------|------|------|
| `POST` | `/tasks/{id}/images` | 单组图片上传（group=A 或 B） |
| `POST` | `/tasks/{id}/images/batch-append` | 批量追加（A/B 两组同时） |
| `POST` | `/tasks/{id}/images/import-zip` | ZIP 压缩包导入 |
| `DELETE` | `/tasks/{id}/images/{img_id}` | 删除单张图片 |
| `PATCH` | `/tasks/{id}/images/reorder` | 重排序（影响配对关系） |

#### ZIP 导入格式

```
upload.zip
  └── root/（名称随意）
      ├── 原图/
      │   ├── img_001.png
      │   └── img_002.png ...
      └── 对比图/
          ├── img_001.png
          └── img_002.png ...
```

- 只接受 `.zip` 格式
- 支持有/无根目录包裹两种结构
- 文件名自动 UTF-8/GBK 解码（兼容 macOS/Windows 压缩）
- 两组按文件名字母序配对，数量可不相等

### 5.3 差异接口

| 方法 | 路径 | 描述 |
|------|------|------|
| `GET` | `/tasks/{id}/diff-status` | 差异计算进度 |
| `GET` | `/tasks/{id}/diff/{pair_index}` | 指定对的差异结果 |
| `POST` | `/tasks/{id}/diff/compute` | 手动触发（全量重算） |

### 5.4 报告接口

| 方法 | 路径 | 描述 |
|------|------|------|
| `GET` | `/tasks/{id}/report` | 生成并返回 HTML 对比报告文件流 |

**响应 Headers**：
- `Content-Type: text/html; charset=utf-8`
- `Content-Disposition: attachment; filename="report_{task_name}.html"`
- `X-Oversized: true` — 图片总大小 ≥ 50MB 时标记（图片改用 MinIO URL 直连）

**报告内容**：
- 任务基本信息、统计卡（总数/正常/异常/待计算）
- 配对列表：状态徽章（正常/异常）+ 文件名 + 缩略图 + 相似度数值
- 点击行展开：A/B 原图两列对比（不含差异标注图）
- 图片 < 50MB 时 Base64 内嵌（离线可用）；≥ 50MB 时 URL 直连（需 MinIO 在线）

## 7. 对比算法详解

### 7.1 四种策略

| 策略 | 名称 | 核心特点 | 典型场景 |
|------|------|---------|---------|
| `balanced` | 标准模式 | 综合感知像素差异与结构 | App UI 截图、通用场景 |
| `document` | 文档对比 | LANCZOS4 对齐 + 低阈值（thr=5） | PDF/文档导出对比，含 DPI 差异检测 |
| `structural` | 结构探测 | 4×4 大分块，侧重宏观布局变化 | 页面元素增删、布局偏移 |
| `pixel_exact` | 像素对比 | 全图无掩码，极严格（thr=3） | 验证两图完全一致（CI 场景） |

### 7.2 算法流程

```
图A + 图B
  ↓
① 尺寸对齐
  ├── 比例差 < 10%: INTER_AREA 缩放（或 LANCZOS4，视策略）
  ├── 比例差 10~30%: 缩放 + size_warning
  └── 比例差 > 30%: ORB 特征点配准 → 失败降级缩放 + size_warning
  ↓
② 生成信息区掩码（Laplacian 梯度 > threshold，膨胀 7×7）
  （pixel_exact 策略不使用掩码，全图参与）
  ↓
③ 四项指标计算（各有权重）
  ↓
④ 加权几何平均 → diff_score（0~1）
  ↓
⑤ diff_score >= 0.75 → is_similar=true（相同）
   diff_score  < 0.75 → is_similar=false（差异）
  ↓
⑥ 差异区域标注 → 差异图上传 MinIO
```

### 7.3 四项指标

| 指标 | 作用 | 核心实现 |
|------|------|---------|
| **掩码差异面积比**（主力） | 统计信息区内有多少比例像素变化了 | `(diff[mask>0] > thr).sum() / mask.sum()` |
| **掩码分块 SSIM** | 8×8 分块，取最差 1/3，跳过空白格 | `skimage.ssim` 分块计算 |
| **全局 SSIM** | 整体结构相似度（辅助，低权重） | `skimage.ssim` 全图 |
| **HSV 颜色直方图** | 颜色主题变化感知 | Bhattacharyya 距离 |

### 7.4 信息区掩码原理

```
原始灰度图
  → Laplacian 梯度（|∇²I|）
  → 阈值过滤（梯度 > 8，文档模式 > 4）
  → 7×7 椭圆核膨胀（覆盖文字周边区域）
  → 二值掩码（1=有内容，0=空白背景）

效果：手机截图信息区约 5%，PDF文档约 13%
      避免 95% 空白背景稀释差异指标，使分数更精准
```

### 7.5 相似度阈值

```
SIMILARITY_THRESHOLD = 0.75

score ≥ 0.75 → is_similar = True  （相同）
score <  0.75 → is_similar = False （差异显著）
```

### 7.6 实测效果参考

| 场景 | 算法 | 得分 |
|------|------|------|
| 完全相同 | 任意 | 100% |
| 同 App 不同页面（明显内容差异） | balanced | 71.4% → 差异 |
| 同 PDF 不同平台导出（渲染细节差） | balanced | 82.4% → 相同 |
| 同 PDF 不同平台导出 | document | 78.4% → 相同 |
| 同内容 DPI 缩小 2× | document | 64.3% → 差异（DPI 变化被检测到）|

## 10. 开发规范与已知问题记录

### 10.1 后端规范

#### 所有路由必须使用 `async def`

同步 `def` 路由在并发轮询时会占满 uvicorn 线程池，导致请求排队响应变慢。

```python
# ✅ 正确
@router.get("/tasks")
async def list_tasks(...): ...

# ❌ 错误
@router.get("/tasks")
def list_tasks(...): ...
```

#### Query 参数校验不使用 `pattern`，手动校验

`pattern` 不允许空字符串，前端 Tab 切换时传 `status=""` 触发 422。

```python
# ✅ 正确：手动处理，空字符串视为 None
async def list_tasks(status: Optional[str] = Query(None)):
    status_filter = status.strip() if status else None
    if status_filter and status_filter not in {"draft", "active", "completed"}:
        raise HTTPException(400, "status 非法")
```

#### pyproject.toml 必须使用 setuptools，不用 hatchling

后端代码平铺在 `backend/` 根目录，无同名子包，hatchling 找不到可打包文件。

```toml
[build-system]
requires = ["setuptools>=68"]
build-backend = "setuptools.build_meta"

[tool.setuptools]
packages = []
py-modules = []

[dependency-groups]    # 注意：tool.uv.dev-dependencies 已弃用
dev = ["pytest>=8", "httpx>=0.27"]
```

#### 路由层手工构造响应必须同步字段

新增模型字段后，所有 `_to_task_detail` / `_to_task_list_item` 等工厂函数必须同步，否则新字段在响应中静默返回默认值（如 `pair_mode`、`diff_algo` 曾遇到此问题）。

#### uvicorn 环境下日志不使用 `logging`，直接写文件

uvicorn 会多次重置 root logger，导致 `logging.getLogger` 的 handler 被覆盖。差异计算日志使用直接文件追加（`diff_service.py` 中的 `diff_log()`），按天滚动。

### 10.2 前端规范

#### Axios 请求必须过滤空参数

```typescript
function cleanParams(params?: Record<string, any>) {
  if (!params) return undefined
  return Object.fromEntries(
    Object.entries(params).filter(([, v]) => v !== undefined && v !== null && v !== '')
  )
}
client.get('/tasks', { params: cleanParams(params) })
```

#### 轮询策略规范

```typescript
// ✅ 先查一次，只在有任务时才启动轮询
diffApi.status(taskId).then((s) => {
  if (s.pending > 0 || s.running > 0) startPoll()
})

// 幂等启动（已有 timer 不重复创建）
const startPoll = useCallback(() => {
  if (timerRef.current) return
  timerRef.current = setInterval(async () => {
    const s = await diffApi.status(taskId)
    setDiffStatus(s)
    if (s.pending === 0 && s.running === 0) stopPoll()
  }, 5000)   // 5 秒间隔
}, [taskId, stopPoll])

// cleanup
useEffect(() => { ...; return stopPoll }, [currentTask?.id])
```

#### useCallback 定义顺序依赖

`useCallback` 内部引用其他 `useCallback` 时，被引用的必须先定义，否则运行时为 `undefined` 导致白屏：

```typescript
// ✅ 正确顺序
const stopPoll  = useCallback(...)    // 先定义
const startPoll = useCallback(...)    // 再定义（引用 stopPoll）
const handleRecompute = useCallback(..., [stopPoll, startPoll])  // 最后

// ❌ 错误：handleRecompute 定义在 stopPoll/startPoll 之前
```

#### 对比页图片展示方案（zoom + 居中）

```tsx
    }} />
```


#### 轮询 setState 前先比对数据变化

```typescript
const s = await diffApi.status(taskId)
const prev = diffStatusRef.current
if (!prev || prev.done !== s.done || prev.running !== s.running) {
  diffStatusRef.current = s

```

### 10.3 Shell 脚本规范

**Shell 脚本字符串内含变量时，所有括号必须用 ASCII `()`，绝不使用全角 `（）`。**

```bash
# ✅ 正确
log_error "端口 $port 已被占用 ($name)"

# ❌ 错误 —— bash 将 "$name）" 整体当变量名，unbound variable
log_error "端口 $port 已被占用（$name）"
```

此问题在 `install.sh`、`start.sh`、`stop.sh` 中均曾出现，已全部修复。

### 10.4 ZIP 导入规范

- Python `zipfile.ZipFile.namelist()` 返回的中文文件名是 CP437 乱码
- 正确做法：`.encode('cp437')` 反转字节后按 UTF-8/GBK 解码
- 目录结构检测不依赖目录条目（macOS zip 不生成目录条目），改为从文件路径推断

### 10.5 前端新增功能时必须同步补 import

**问题**：向 `CompareListPage.tsx` 新增 `handleExportReport` 时用了 `toast.loading/success/error`，但没有添加 `import { toast } from 'sonner'`，运行时报 `ReferenceError: toast is not defined`，表现为按钮点击无响应。

**为什么 TS 没报错**：`toast` 不是全局声明，但 tsc 在 async 函数体 + promise chain 中有时不立即报错。

**规范**：向已有文件新增功能块时，先检查依赖的工具函数/组件是否已 import；常见遗漏：`toast`（sonner）、`useCallback/useRef/memo`（react）、图标（lucide-react）；改完必跑 `npx tsc --noEmit`。

### 10.6 diff_results.pair_index 不可使用旧缓存值，必须同步更新

**问题**：`_run_diff_internal` 遍历配对时，若 `diff_results` 表中已存在对应记录（`existing`），直接复用旧记录不更新 `pair_index`。当图片重排序或配对模式改变后，同一对图片的 `pair_index` 发生变化，旧记录的 `pair_index` 错位，导致前端按索引查询时找不到，显示「待计算」。

**修复**：
```python
if existing:
    diff_record = existing
    # 同步修正 pair_index（旧记录可能因重排序而错位）
    if diff_record.pair_index != pair_index:
        diff_record.pair_index = pair_index
```

**兜底**：`GET /diff/{pair_index}` 按索引找不到时，再按 `(image_a_id, image_b_id)` 查，找到后自动修正 `pair_index` 并返回。
